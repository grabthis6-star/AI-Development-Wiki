# GuideFlow 개발 기술

기술 이름을 외우기 위한 문서가 아닙니다. GuideFlow의 실제 기능을 만들 때 무엇이 필요했고, 어떤 문제가 있었으며, 다음 프로그램에서 어떻게 재사용할 수 있는지 설명합니다.

## 1. 캡처 이미지 붙여넣기

### 무엇인가
Windows 캡처나 웹에서 복사한 이미지를 Ctrl+V로 프로그램에 바로 추가하는 기능입니다.

### 왜 필요한가
GuideFlow의 핵심은 생각의 흐름을 끊지 않는 것입니다. 파일 저장과 가져오기 창을 반복하면 5분 스케치 도구라는 목적이 무너집니다.

### 사용 기술
- 브라우저의 paste 이벤트
- Clipboard API
- 클립보드 항목의 이미지 MIME 타입 확인
- 이미지 파일을 화면에서 볼 수 있게 바꾸는 Object URL 또는 FileReader

### 실제로 주의할 문제
텍스트가 들어 있는 클립보드와 이미지가 들어 있는 클립보드를 구분해야 합니다. 같은 붙여넣기라도 선택된 이미지가 있으면 교체, 없으면 새 이미지 추가로 동작할 수 있으므로 현재 선택 상태가 중요합니다.

### 다른 프로그램에서 재사용
무드보드, 카드뉴스 제작기, 상품 이미지 정리, 디자인 레퍼런스 보드에 그대로 적용할 수 있습니다.

## 2. 붙여넣기로 이미지 교체

### 무엇인가
기존 이미지의 위치와 크기는 유지하면서 이미지 내용만 새 캡처로 바꾸는 기능입니다.

### 왜 필요한가
새 이미지를 추가한 뒤 위치와 크기를 다시 맞추는 과정을 없애 작업 속도를 높입니다.

### 쉽게 비유하면
액자는 그대로 두고 사진만 갈아 끼우는 것입니다.

### 실제로 주의할 문제
현재 선택한 요소가 이미지인지 확인하지 않으면 엉뚱한 장면에 새 이미지가 추가될 수 있습니다. 새 이미지 추가와 선택 이미지 교체를 별도 명령으로 구분하는 것도 안전한 방법입니다.

### 다른 프로그램에서 재사용
프로필 이미지, 상품 사진, 프레젠테이션 이미지와 카드 템플릿 교체 기능에 사용할 수 있습니다.

## 3. 이미지 이동과 크기 조절

### 무엇인가
사용자가 보드 위의 이미지를 직접 끌어 이동하고 크기를 조정하는 기능입니다.

### 왜 필요한가
GuideFlow는 정확한 디자인보다 전체적인 구도와 느낌을 빠르게 확인하는 도구이므로 숫자 입력보다 직접 조작이 잘 맞습니다.

### 사용 기술
- Pointer Event
- 드래그 시작, 이동, 종료 상태
- 포인터 시작 좌표와 이미지 시작 위치의 차이 계산
- 보드 경계와 최소 크기 제한
- pointer capture를 이용한 안정적인 드래그

### 실제로 주의할 문제
별도의 이미지 조정 화면으로 이동하면 전체 장면을 보며 수정하기 어렵습니다. 메인 보드에서 선택과 조정을 함께 처리하는 편이 GuideFlow의 목적에 맞습니다.

### 다른 프로그램에서 재사용
칸반 보드, 슬라이드, 스티커 배치, 지도 마커와 간단한 디자인 도구에 사용할 수 있습니다.

## 4. 장면 데이터와 고유 ID

### 무엇인가
각 장면의 이미지, 시간, 메모와 순서를 하나의 데이터로 관리하는 방식입니다.

### 왜 필요한가
장면을 삭제하거나 순서를 바꾸면 화면상의 1번, 2번 번호는 달라집니다. 내부 고유 ID까지 순번으로 사용하면 다른 장면을 수정할 수 있습니다.

### 핵심 원칙
- 화면 순번과 내부 ID를 분리합니다.
- 현재 선택된 장면 ID를 별도로 관리합니다.
- 장면을 복제할 때 새 ID를 발급합니다.
- 정렬 후 화면만이 아니라 데이터 배열도 같은 순서로 갱신합니다.

### 다른 프로그램에서 재사용
슬라이드 제작기, 설문 문항, 강의 목차, 쇼츠 장면과 카드 목록 관리에 적용할 수 있습니다.

## 5. 보드 중심 편집 화면

### 무엇인가
보드 보기와 편집 보기를 따로 두지 않고 전체 장면을 보면서 선택한 이미지를 바로 조정하는 화면 구조입니다.

### 왜 필요한가
GuideFlow 사용자는 정교한 편집보다 전체 흐름을 빠르게 잡고 싶습니다. 모드 전환과 별도 설정 창이 많아질수록 기존 편집 프로그램과 비슷하게 무거워집니다.

### 설계 원칙
- 가장 자주 쓰는 기능을 보드에 노출합니다.
- 설명용으로 큰 공간을 차지하는 안내 상자는 첫 사용 이후 줄입니다.
- 선택된 장면과 이미지 상태는 명확히 표시합니다.
- 버튼을 추가하기 전에 기존 기능과 묶거나 제거할 수 있는지 봅니다.

### 다른 프로그램에서 재사용
초보자용 제작 도구에서는 여러 모드보다 한 화면에서 직접 조작하는 구조가 효과적입니다.

## 6. 장면별 시간과 재생

### 무엇인가
각 장면이 화면에 표시되는 시간을 저장하고, 재생 버튼을 누르면 해당 시간만큼 순서대로 보여주는 기능입니다.

### 사용 기술
- 장면별 duration 값
- 현재 재생 인덱스
- setTimeout 또는 시간 기반 재생 루프
- 재생 시작, 중지와 종료 상태 정리

### 마지막 장면 오류
다음 장면으로 이동하는 로직만 구현하면 마지막 장면은 다음 대상이 없어 즉시 종료될 수 있습니다.

### 해결 원칙
마지막 장면도 자신의 duration 동안 보여준 뒤 전체 재생을 종료해야 합니다. 화면 재생뿐 아니라 내보내기 결과에서 마지막 장면 길이를 별도로 확인합니다.

### 다른 프로그램에서 재사용
슬라이드쇼, 디지털 사이니지, 스토리보드와 자동 재생 프레젠테이션에 적용할 수 있습니다.

## 7. 내보내기와 결과 검증

### 무엇인가
보드에 만든 장면 순서와 시간을 다른 사람에게 보여주거나 편집 참고용으로 사용할 수 있는 결과물로 만드는 기능입니다.

### 왜 별도 검증이 필요한가
편집 화면에서 정상으로 보여도 내보내기 과정에서는 마지막 장면, 이미지 로딩, 화면 비율과 시간이 다르게 처리될 수 있습니다.

### 검증 순서
1. 장면 수가 같은지 확인합니다.
2. 장면 순서가 같은지 확인합니다.
3. 각 장면의 시간이 적용됐는지 확인합니다.
4. 마지막 장면 유지 시간을 확인합니다.
5. 쇼츠 9:16 비율과 이미지 잘림을 확인합니다.
6. 생성된 결과물을 처음부터 끝까지 직접 재생합니다.

## 8. 모바일 버튼과 반응형 UI

### 무엇인가
화면 크기가 작아져도 버튼, 글자와 보드가 겹치거나 밖으로 튀어나오지 않게 만드는 작업입니다.

### 실제로 발생한 문제
버튼 글자가 컨테이너 밖으로 나가고 버튼 배열이 깨지는 문제가 반복됐습니다.

### 해결 원칙
- 버튼 영역에 flex-wrap을 사용합니다.
- 고정 너비를 줄이고 최소 너비와 유동 너비를 조합합니다.
- 긴 버튼 문구를 줄입니다.
- 모바일에서는 중요하지 않은 설명을 숨깁니다.
- 실제 모바일 폭에서 스크린샷과 클릭으로 확인합니다.

### 다른 프로그램에서 재사용
도구형 웹사이트에서는 데스크톱 축소판이 아니라 핵심 작업 순서에 맞춘 모바일 배치가 필요합니다.

## 9. 작업 상태 저장

### LocalStorage
간단한 설정과 작은 JSON 데이터를 브라우저에 보관할 수 있습니다. 빠른 프로토타입에는 좋지만 큰 이미지와 중요한 영구 프로젝트에는 적합하지 않습니다.

### 저장 방식을 고를 때 질문
- 새로고침 후만 유지하면 되는가?
- 다른 컴퓨터에서도 열어야 하는가?
- 다른 사람에게 전달해야 하는가?
- 이미지 원본까지 보관해야 하는가?

필요 범위가 커지면 파일 저장, IndexedDB 또는 서버 저장을 검토합니다.

## 10. GitHub와 Vercel 배포

### GitHub
원본 코드와 변경 이력을 저장합니다. 어떤 변경이 문제를 만들었는지 커밋 단위로 비교할 수 있습니다.

### Vercel
GitHub의 웹 프로젝트를 실제 주소로 배포합니다.

### 실제로 중요했던 문제
수정한 index.html과 실제 빌드에서 사용하는 dashboard-v2.html이 달라 사이트가 바뀌지 않은 적이 있었습니다.

### 검증 원칙
1. 실제 빌드 진입 파일을 확인합니다.
2. GitHub 최신 커밋을 확인합니다.
3. Vercel 배포 상태를 확인합니다.
4. 공개 주소를 새로 열어 화면을 확인합니다.
5. 버튼과 핵심 기능을 직접 사용합니다.
