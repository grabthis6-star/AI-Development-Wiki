# 위키 운영 규칙

## 목표

이 위키를 일회성 설명 사이트가 아니라 여러 프로젝트에서 계속 사용하는 개인 개발 지식 시스템으로 유지합니다.

## 무엇을 기록할 것인가

다음 중 하나에 해당하면 새 문서 또는 기존 문서 업데이트 대상입니다.

- 처음 알게 된 개발 도구
- 두 번 이상 설명할 가능성이 있는 개념
- 30분 이상 걸린 오류
- 다음 프로젝트에서도 사용할 체크리스트
- 자주 쓰는 AI 요청 문구
- 중요한 구조 결정과 그 이유
- 실패한 시도와 다시 하지 말아야 할 이유

## 문서 기본 형식

1. 한 줄 설명
2. 왜 필요한가
3. 쉽게 비유하면
4. 실제 프로젝트에서는 어떻게 사용했는가
5. 헷갈리는 개념
6. 이것만 기억할 한 줄
7. 관련 문서

모든 항목이 필요하지는 않지만, 초보자가 다시 읽어도 이해할 수 있어야 합니다.

## 기록 위치

- 웹사이트: 사람이 읽고 검색하는 화면
- GitHub `docs/`: 장기 보관과 AI 참고용 원본 문서
- README: 프로젝트 전체 안내와 주요 문서 입구
- Git 커밋: 언제 무엇을 바꿨는지 남기는 변경 기록

## 업데이트 흐름

```text
프로젝트에서 새로운 경험 발생
        ↓
재사용 가치 판단
        ↓
위키 문서 작성 또는 수정
        ↓
GitHub main에 기록
        ↓
Vercel 자동 배포
        ↓
실제 사이트 확인
```

## 문서 품질 기준

- 어려운 용어만 나열하지 않습니다.
- 사실과 추측을 구분합니다.
- 확인하지 않은 저장소 상태를 기록하지 않습니다.
- 특정 프로젝트 사례와 일반 규칙을 구분합니다.
- 오래된 내용은 삭제보다 수정 기록을 남기며 업데이트합니다.
- 비밀번호, 토큰, 개인 정보는 기록하지 않습니다.

## 백업 원칙

핵심 문서는 웹 화면에만 두지 않고 `docs/`에도 Markdown으로 보관합니다.

장기적으로 다음 내보내기 기능을 추가합니다.

- Markdown 전체 백업
- Word 내보내기
- PDF 내보내기
- JSON 데이터 백업

## 정기 점검

프로젝트 하나가 끝날 때 다음을 확인합니다.

- 새로 배운 개념이 있는가?
- 오래 걸린 오류가 있는가?
- 다음에도 사용할 프롬프트가 있는가?
- 기존 위키 내용 중 틀리거나 부족한 것이 있는가?
- 시작 문서의 작업 규칙을 바꿔야 하는가?

## 이것만 기억하자

> 프로젝트가 성장할수록 위키도 함께 성장해야 다음 프로젝트의 출발점이 높아집니다.
