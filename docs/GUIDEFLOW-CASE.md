# GuideFlow 사례

## 프로젝트 성격

GuideFlow는 영상 편집기가 아니라 장면을 계획하고 정리하는 스토리보드 도구입니다.

## 사용한 기본 구조

```text
React + TypeScript + Vite
        ↓
GitHub main 브랜치에 원본 저장
        ↓
Vercel 자동 배포
        ↓
실제 사이트에서 사용자가 확인
```

## 확인된 주요 기능

- 장면 번호 표시
- 장면 복제
- `.guideflow` 프로젝트 저장과 불러오기
- 기본 이미지 교체
- 장면 카드와 액션 배치 개선

## 검증된 개발 방식

1. 사용자가 필요한 기능을 설명합니다.
2. AI가 관련 저장소와 파일을 확인합니다.
3. 기능을 구현하고 GitHub에 기록합니다.
4. Vercel이 자동 배포합니다.
5. 사용자가 실제 사이트에서 확인합니다.
6. 문제가 있으면 화면과 재현 순서를 전달합니다.
7. 해결 후 다음 기능으로 이동합니다.

## 잘 작동한 점

- 기능을 작은 단위로 나누면 문제 원인을 찾기 쉬웠습니다.
- 실제 배포 사이트를 기준으로 판단해 개발자 컴퓨터에서만 작동하는 오류를 줄였습니다.
- GitHub 변경 기록 덕분에 무엇을 언제 바꿨는지 추적할 수 있었습니다.
- 사용자가 개발 용어를 몰라도 화면과 원하는 결과를 설명해 진행할 수 있었습니다.

## 주의할 점

- 코드가 GitHub에 들어갔다는 말은 실제 확인이 필요합니다.
- Vercel의 이전 배포 화면과 최신 배포 화면을 혼동하지 않아야 합니다.
- 한 기능을 고치면서 다른 버튼의 위치나 동작을 깨뜨리지 않는지 확인해야 합니다.
- 플러그인이나 임시 해결책이 많아지면 프로젝트 구조가 복잡해질 수 있습니다.

## AI Development Wiki가 생긴 이유

GuideFlow 개발 과정에서 중요한 내용이 긴 채팅에 흩어졌습니다. 새 채팅이나 새 프로젝트에서는 같은 설명을 다시 해야 했습니다.

이 문제를 해결하기 위해 다음 내용을 장기적으로 보관하는 별도의 위키를 만들었습니다.

- 개발 도구의 쉬운 설명
- 사용자와 AI의 협업 규칙
- 자주 쓰는 프롬프트
- 문제 원인과 해결법
- 프로젝트 시작 체크리스트
- 실제 프로젝트 사례

## 다음 프로젝트에 재사용할 규칙

- 프로젝트의 목적을 먼저 한 문장으로 정합니다.
- 최소 기능부터 실제 주소에서 작동하게 만듭니다.
- 위험한 기능은 하나씩 확인합니다.
- 단순 문서와 안전한 개선은 여러 단계 묶어 진행할 수 있습니다.
- 완료 여부는 실제 결과로 판단합니다.
- 오래 걸린 문제는 반드시 위키에 기록합니다.

## 이것만 기억하자

> GuideFlow의 가장 큰 결과는 앱 하나뿐 아니라, AI와 프로그램을 만드는 검증된 작업 방식을 얻은 것입니다.
