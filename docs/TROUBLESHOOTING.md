# 문제 해결 기록법

## 목적

오류를 한 번 고치는 데서 끝내지 않고, 다음 프로젝트에서도 같은 문제를 빠르게 해결할 수 있도록 기록합니다.

## 기록 순서

### 1. 정상이어야 하는 모습

사용자가 기대한 결과를 적습니다.

### 2. 실제 현상

화면, 오류 문구, 작동하지 않는 기능을 그대로 적습니다.

### 3. 재현 순서

문제를 다시 발생시키는 순서를 번호로 적습니다.

### 4. 직전 변경

문제가 나타나기 전에 무엇을 바꿨는지 적습니다.

### 5. 시도한 것

효과가 없었던 시도도 기록합니다. 같은 실패를 반복하지 않게 해 줍니다.

### 6. 실제 원인

추측이 아니라 확인된 원인을 적습니다.

### 7. 해결 방법

어떤 파일과 설정을 어떻게 바꿨는지 적습니다.

### 8. 검증

어떤 방법으로 문제가 해결됐음을 확인했는지 적습니다.

### 9. 다음 체크리스트

다음에 같은 현상이 나타나면 먼저 확인할 항목을 적습니다.

## 오류 자료로 유용한 것

- 전체 오류 문구
- 화면 캡처
- 발생한 사이트 주소
- Vercel Build Logs
- GitHub의 최근 변경 기록
- 문제가 발생하는 정확한 클릭 순서
- 사용한 브라우저와 기기

## AI에게 전달하는 오류 수정 문구

```text
정상이어야 하는 모습:
[작성]

실제 현상:
[작성]

재현 순서:
1. ...
2. ...

오류 문구 또는 화면:
[붙여 넣기]

최근 변경:
[알고 있다면 작성]

추측으로 여러 곳을 동시에 바꾸지 말고, 관련 파일과 실제 원인을 먼저 확인한 뒤 가장 작은 수정으로 고쳐줘. 수정 후 확인 방법도 알려줘.
```

## 이번 위키의 첫 배포에서 배운 것

Vercel 프로젝트 연결이 성공해도 실제 사이트가 정상이라는 뜻은 아니었습니다. 빈 저장소를 먼저 연결한 뒤 프레임워크와 결과 폴더 설정이 맞지 않아 404와 빌드 오류가 발생했습니다.

해결 과정에서는 외부 의존성이 없는 단순한 정적 빌드로 먼저 배포 경로를 정상화했습니다.

### 다음에 먼저 확인할 것

1. `package.json`의 build 명령
2. Vercel의 Framework Preset
3. Output Directory
4. 최신 배포가 Ready인지
5. 실제 사이트 주소가 열리는지
6. 최신 GitHub 커밋이 배포에 사용됐는지

## 이것만 기억하자

> 현상 → 재현 → 원인 → 최소 수정 → 실제 검증 → 다음 체크리스트
