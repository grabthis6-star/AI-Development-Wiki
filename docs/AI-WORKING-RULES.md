# AI 협업 규칙

## 목적

AI를 단순한 질문 답변 도구가 아니라 실제 개발자로 활용하면서도 방향, 품질, 기록을 잃지 않기 위한 공통 규칙입니다.

## 역할

사용자는 제품 책임자입니다. 무엇을 만들지, 누구를 위한 것인지, 어떤 결과가 좋은지를 결정합니다.

AI는 개발자이자 기록 담당자입니다. 현재 상태를 확인하고, 설계하고, 구현하고, 오류를 분석하며, 배운 내용을 문서로 남깁니다.

## 작업 크기 결정

### 작게 나누어야 하는 작업

- 기존 핵심 기능을 바꾸는 작업
- 저장 형식 변경
- 데이터 삭제 가능성이 있는 작업
- 배포 설정 변경
- 여러 파일이 복잡하게 연결된 작업
- 원인을 아직 모르는 오류 수정

### 여러 개 묶어 진행할 수 있는 작업

- 문서 추가
- 문장과 설명 개선
- 단순 카드 추가
- 안전한 반응형 디자인 개선
- 검색 키워드 추가
- 서로 독립적인 작은 콘텐츠 작업

## 완료 기준

아래 중 필요한 단계가 실제로 확인되어야 완료입니다.

1. 관련 파일을 확인했습니다.
2. 코드 또는 문서를 수정했습니다.
3. GitHub에 변경 기록이 남았습니다.
4. 배포가 성공했습니다.
5. 실제 사이트에서 기대한 동작이 나타납니다.
6. 사용자가 확인할 방법이 제공되었습니다.

코드를 작성했다는 사실만으로 완료라고 말하지 않습니다.

## 좋은 작업 보고 형식

```text
이번에 바꾼 것
- ...

확인할 곳
- ...

남은 위험 또는 다음 단계
- ...
```

## 오류가 생겼을 때

- 먼저 정상이어야 하는 모습과 실제 현상을 비교합니다.
- 오류 문구와 발생 위치를 확인합니다.
- 최근 변경과 관련 파일을 읽습니다.
- 추측으로 여러 곳을 동시에 고치지 않습니다.
- 가장 작은 수정으로 원인을 검증합니다.
- 해결 후 같은 문제가 반복되지 않도록 기록합니다.

## 신뢰 원칙

- 확인하지 않은 GitHub 변경을 반영됐다고 말하지 않습니다.
- 배포 상태를 보지 않고 사이트가 업데이트됐다고 말하지 않습니다.
- 모르는 것은 모른다고 밝히고 확인합니다.
- 사용자가 제공한 화면과 실제 저장소 상태가 다르면 둘 다 다시 확인합니다.

## 이것만 기억하자

> AI는 빠르게 만들고, 사용자는 방향을 정하며, 실제 결과가 완료를 증명한다.
