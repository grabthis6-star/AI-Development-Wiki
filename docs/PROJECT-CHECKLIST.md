# 프로젝트 시작 체크리스트

## 1. 목적

- 누가 사용할 것인가?
- 어떤 상황에서 사용할 것인가?
- 무엇을 더 쉽고 빠르게 만들 것인가?

한 문장으로 설명할 수 있을 때까지 줄입니다.

## 2. 첫 버전

처음부터 모든 기능을 만들지 않습니다.

- 반드시 필요한 기능 1~3개
- 사용자가 처음부터 끝까지 완료할 수 있는 핵심 흐름
- 없어도 첫 사용이 가능한 장식 기능은 뒤로 미루기

## 3. 이름과 저장소

- 프로젝트 이름
- GitHub 저장소 이름
- 공개 또는 비공개 여부
- 기본 브랜치 확인
- README 작성

## 4. 기술 선택

기술 이름보다 필요한 결과를 먼저 정합니다.

- 단순한 정보 사이트인가?
- 사용자 입력과 저장이 필요한가?
- 로그인과 데이터베이스가 필요한가?
- 휴대전화에서도 사용해야 하는가?
- Word, PDF, 이미지 같은 파일 내보내기가 필요한가?

## 5. 배포

웹 프로젝트라면 다음을 확인합니다.

- GitHub와 Vercel 연결
- 기준 브랜치 확인
- 빌드 명령 확인
- 결과 폴더 확인
- 실제 사이트 주소 열기
- 새 변경이 자동 배포되는지 확인

## 6. 작업 규칙

- 위험한 기능은 하나씩 진행
- 안전한 문서 작업은 묶어서 진행 가능
- 변경 기록 남기기
- 실제 사이트에서 테스트
- 확인되지 않은 작업은 완료 처리하지 않기

## 7. 기록

아래 내용이 생기면 위키에 추가합니다.

- 처음 알게 된 도구
- 중요한 구조 결정
- 오래 걸린 오류
- 여러 번 재사용할 명령어
- 다음 프로젝트에서도 적용할 작업 방식
- 실패한 시도와 그 이유

## 최종 점검

- [ ] 목적을 한 문장으로 설명할 수 있다.
- [ ] 첫 버전 기능이 1~3개로 정리됐다.
- [ ] GitHub 저장소가 있다.
- [ ] 실제 배포 주소가 있다.
- [ ] 변경과 배포 흐름을 확인했다.
- [ ] 새 채팅에 전달할 시작 문구가 준비됐다.
- [ ] 새로 배운 내용을 기록할 장소가 있다.

## 이것만 기억하자

> 목적 → 최소 기능 → GitHub → 배포 → 실제 확인 → 기록
