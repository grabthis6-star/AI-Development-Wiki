# AI와 프로그램 만드는 실전 운영 가이드

커뮤니티 대화에서 반복된 실제 어려움을 바탕으로, 사용자가 준비하고 AI에게 맡기고 직접 확인해야 할 일을 정리했습니다. 제품 성능·가격·출시 소문은 사실로 사용하지 않으며 최신 조건은 공식 문서에서 다시 확인합니다.

## 실전 운영 가이드

### 🧱 AI 도구의 층을 구분하기
서비스·모델·하네스·앱·CLI·스킬·MCP를 한 장의 관계도로 이해합니다.
- 필요한 이유: 제품 이름만 비교하면 같은 모델을 다른 작업 환경에서 쓰는 차이를 놓치기 쉽습니다.
- 사용자가 할 일: 만들 작업을 한 문장으로 적기 → 파일·브라우저·외부 서비스 중 필요한 능력 표시 → 그 능력을 제공하는 작업 환경과 연결 기능 선택
- AI가 할 일: 필요한 모델보다 필요한 작업 능력을 먼저 분석 → 최소한의 하네스와 연결만 제안 → 각 연결의 권한과 검증 방법 설명
- 확인 항목: 모델과 앱을 같은 것으로 설명하지 않았다. / 필요하지 않은 MCP·플러그인을 추가하지 않았다.
- 복사 요청문:

> 이번 작업에 필요한 능력을 모델, 작업 환경, 사용 방식, 확장 기능, 외부 서비스로 나눠 설명하고 최소 구성만 추천해.

### ⌨️ 초보자가 CLI를 안전하게 시작하기
명령어를 외우지 않고 AI가 설치·실행·검증을 담당하게 합니다.
- 필요한 이유: Whisper·FFmpeg 같은 로컬 도구는 터미널 설치가 필요할 수 있지만 사용자가 개발자가 될 필요는 없습니다.
- 사용자가 할 일: 운영체제와 만들 결과 알려주기 → 설치 전 용량·권한·제거 방법 확인 → 실행 결과 전체를 AI에게 전달
- AI가 할 일: 현재 설치 상태 먼저 확인 → 필요한 도구만 한 단계씩 설치 → 버전과 짧은 샘플로 정상 작동 검증
- 확인 항목: 설치 이유를 이해했다. / 정상 버전과 샘플 결과를 확인했다. / 다른 컴퓨터에 필요한 조건을 기록했다.
- 복사 요청문:

> 나는 개발 경험이 없어. 현재 컴퓨터 상태를 먼저 확인하고 꼭 필요한 도구만 설치해. 각 도구의 역할, 용량·권한·제거 방법과 정상 확인 결과를 쉬운 말로 알려줘.

### 🧠 세션과 컨텍스트 가볍게 유지하기
작업 단위로 새 세션을 열고 프로젝트 문서로 맥락을 이어갑니다.
- 필요한 이유: 긴 대화와 많은 도구가 누적되면 오래된 지시가 충돌하고 원인 파악이 어려워질 수 있습니다.
- 사용자가 할 일: 완료된 작업을 Git에 저장 → 컨텍스트 문서에 현재 상태 기록 → 새 작업에는 이번 목표와 유지 기능만 전달
- AI가 할 일: 필요한 파일만 읽기 → 오래된 지시와 현재 목표의 충돌 알리기 → 완료 후 변경·검증 결과 요약
- 확인 항목: 새 작업에서 이전 기능을 다시 확인했다. / 사용하지 않는 연결을 껐다. / 현재 목표가 한 문장으로 명확하다.
- 복사 요청문:

> 현재 변경을 먼저 저장하고 컨텍스트 문서를 갱신해. 새 작업에는 이번 목표, 유지 기능, 완료 기준만 남길 수 있게 인수인계문을 만들어줘.

### 🔐 MCP·플러그인 권한 확인하기
많이 연결하기보다 현재 작업에 필요한 최소 능력만 허용합니다.
- 필요한 이유: 외부 연결은 편리하지만 파일·저장소·메일·드라이브에 읽기나 쓰기 권한을 줄 수 있습니다.
- 사용자가 할 일: 출처와 제작자 확인 → 읽기·쓰기 권한 구분 → 삭제·게시·전송은 실행 전 확인하도록 설정
- AI가 할 일: 전송되는 데이터와 권한 범위 설명 → 최소 권한 대안 제안 → 사용하지 않을 때 해제 방법 안내
- 확인 항목: API 키를 채팅이나 저장소에 넣지 않았다. / 외부 전송 데이터가 명확하다. / 삭제·게시에는 사람 확인이 있다.
- 복사 요청문:

> 이 연결을 사용하기 전에 출처, 읽기·쓰기 권한, 외부 전송 데이터, 비밀정보 보관, 해제 방법을 점검해. 필요한 최소 권한만 제안해.

### 🗃️ JSON과 폴더를 중간 결과로 남기기
AI가 바로 최종 결과를 만들게 하지 않고 수정 가능한 구조화 데이터를 먼저 만듭니다.
- 필요한 이유: 도구가 바뀌어도 데이터를 재사용하고 실패한 장면만 다시 만들 수 있습니다.
- 사용자가 할 일: 장면·항목별 필요한 정보 정하기 → 폴더와 파일 이름 규칙 결정 → 중간 결과를 눈으로 확인
- AI가 할 일: JSON 구조와 예시 생성 → 입력·출력 폴더 자동 구성 → 누락·중복 검사 후 다음 도구에 전달
- 확인 항목: 각 항목에 고유 번호가 있다. / 원본과 생성 결과가 분리됐다. / 한 항목만 다시 실행할 수 있다.
- 복사 요청문:

> 최종 결과를 바로 만들지 말고 먼저 재사용 가능한 JSON 구조와 입력·출력 폴더를 설계해. 사람이 중간 결과를 수정한 뒤 다음 단계로 넘기게 해.

### 🎧 영상·음성 결과를 다시 검사하기
생성된 음성을 다시 글자로 바꾸고 원본 대본과 비교해 오류 구간만 찾습니다.
- 필요한 이유: TTS 결과를 끝까지 직접 듣는 부담을 줄이면서 누락·잘못된 발음 후보를 빠르게 찾을 수 있습니다.
- 사용자가 할 일: 원본 대본과 결과 파일 제공 → 허용할 차이와 고유명사 표시 → AI가 찾은 구간을 직접 들어 최종 판단
- AI가 할 일: 음성 추출과 전사 → 원본 대본과 구간별 비교 → 문제 후보와 재생 위치를 보고서로 생성
- 확인 항목: 자동 전사를 정답으로 취급하지 않았다. / 사람이 문제 구간을 실제로 들었다. / 수정할 부분만 다시 생성했다.
- 복사 요청문:

> 영상에서 음성을 추출해 전사하고 원본 대본과 비교해. 누락·반복·발음 오류 후보의 시간 위치를 보고하고, 최종 판단은 내가 직접 들은 뒤 하게 해.

### ☁️ 로컬과 클라우드 선택하기
처음에는 빠르게 시험하고 반복 비용·개인정보·성능 문제가 생길 때 실행 위치를 다시 고릅니다.
- 필요한 이유: 로컬이 항상 싸거나 클라우드가 항상 쉬운 것이 아니며 설치와 장비 비용까지 함께 봐야 합니다.
- 사용자가 할 일: 월 예상 사용량과 보낼 수 없는 데이터 표시 → 컴퓨터 사양과 설치 부담 확인 → 작은 샘플로 품질과 속도 비교
- AI가 할 일: 초기비용·반복비용·속도·개인정보 비교 → 필요 사양과 유지관리 설명 → 가장 작은 검증 방법 제안
- 확인 항목: 커뮤니티 경험을 확정 사실로 쓰지 않았다. / 공식 요구 사양과 가격표를 확인했다. / 실험 비용 한도를 정했다.
- 복사 요청문:

> [기능]을 로컬과 클라우드로 실행할 때 초기비용, 반복비용, 설치 난이도, 속도, 개인정보와 공식 요구 사양을 비교하고 작은 시험부터 제안해.

### 💳 API와 자동화 비용 제한하기
실행 전에 호출 횟수·재시도·월 한도·중지 방법을 정합니다.
- 필요한 이유: 자동화는 사람이 보지 않는 동안 실패 호출과 생성 비용이 반복될 수 있습니다.
- 사용자가 할 일: 한 번·하루·한 달 최대 비용 결정 → 처음에는 3개 이하 샘플로 시험 → 비용 알림과 중지 조건 승인
- AI가 할 일: 예상 호출량과 비용 계산식 제시 → 무한 재시도 방지 → 실행 로그·실패 알림·중지 기능 구현
- 확인 항목: API 키가 안전하게 보관된다. / 재시도 횟수에 제한이 있다. / 예산 초과 시 자동 중지된다.
- 복사 요청문:

> 구현 전에 한 번 실행할 때의 호출 횟수와 비용 계산식을 보여줘. 샘플 3개로 검증하고 일·월 한도, 재시도 제한, 로그와 즉시 중지 방법을 포함해.

## 현장에서 바로 쓰는 팁

### 🔤 이미지와 글자를 따로 만들기
- 핵심: AI 이미지 안에 정확한 한글을 억지로 생성하지 말고, 글자 없는 배경을 만든 뒤 HTML·Canvas·편집 도구로 문구를 입힙니다.
- 실행: 배경 이미지, 표시 문구, 글꼴·위치·색상을 별도 데이터로 관리합니다.
- 주의: 이미지 생성 모델의 글자 품질은 도구와 시점에 따라 달라지므로 공식 기능과 짧은 샘플을 먼저 확인합니다.

### 🧩 템플릿을 먼저 만들고 내용만 교체하기
- 핵심: 매번 전체 화면이나 영상을 새로 만들기보다 검증된 레이아웃 하나를 만든 뒤 이미지·대사·시간만 바꿉니다.
- 실행: 템플릿 ID와 교체 가능한 항목을 JSON에 적고, 새 결과는 데이터만 바꿔 생성합니다.
- 주의: 템플릿이 완성되기 전에 대량 생성을 시작하지 않습니다.

### 📓 피드백을 프로젝트 문서에 누적하기
- 핵심: AI에게 같은 실수를 반복해서 설명하지 않도록 승인된 규칙과 실패 사례를 Markdown 문서에 남깁니다.
- 실행: 피드백을 사실·선호·금지사항·검증법으로 구분해 컨텍스트 문서나 AGENTS.md에 추가합니다.
- 주의: 일회성 감상이나 오래된 규칙까지 모두 넣으면 오히려 충돌하므로 현재도 유효한 내용만 유지합니다.

### 📦 대량 작업은 3개 샘플부터 시작하기
- 핵심: 수백 개를 바로 생성하지 말고 3개 샘플로 품질·시간·비용·파일명을 먼저 검증합니다.
- 실행: 샘플 성공 후 10개, 50개처럼 단계적으로 늘리고 동시 실행 수와 재시도를 제한합니다.
- 주의: 병렬 실행은 속도뿐 아니라 API 제한·메모리·비용도 함께 증가시킬 수 있습니다.

### 🔁 실패 지점부터 다시 시작할 수 있게 만들기
- 핵심: 한 장면의 실패 때문에 전체 작업을 처음부터 다시 돌리지 않도록 각 단계의 상태와 결과를 저장합니다.
- 실행: 항목별 상태를 대기·진행·완료·실패로 기록하고 실패 항목만 재실행하는 버튼이나 명령을 둡니다.
- 주의: 성공 여부를 파일 존재만으로 판단하지 말고 크기·형식·내용도 확인합니다.

### 🧑‍🤝‍🧑 도구마다 잘하는 일을 나누기
- 핵심: 한 AI에게 조사·글쓰기·코딩·브라우저 검증을 전부 맡기기보다 작업에 맞는 도구를 연결합니다.
- 실행: 조사, 구조화, 구현, 렌더링, 검수의 담당과 전달 파일을 먼저 정합니다.
- 주의: 도구가 많아질수록 전달 오류와 권한 위험도 늘어나므로 최소 구성으로 시작합니다.

### 🎯 초벌 자동화와 최종 품질을 분리하기
- 핵심: AI는 반복되는 기본 틀과 문제 후보를 만들고, 사람은 의도·리듬·사실·저작권·최종 사용감을 결정합니다.
- 실행: 자동 생성이 끝나는 지점과 사람이 승인해야 다음 단계로 가는 품질 관문을 명시합니다.
- 주의: 검수 단계를 없앤 완전 자동화를 성공 기준으로 삼지 않습니다.

## 커뮤니티 정보 사용 원칙

- 개인의 성능·토큰·비용 경험을 보편적인 사실로 단정하지 않습니다.
- CAPTCHA·내부 API·심사·안전장치 우회 방법은 안내하지 않습니다.
- 저작권 있는 자료의 무단 다운로드와 재사용을 자동화하지 않습니다.
- 서비스 가격·지원 기능·요구 사양은 공식 문서에서 마지막으로 확인합니다.
