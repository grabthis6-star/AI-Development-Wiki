# GuideFlow 개발 컨텍스트

이 문서는 새 GPT 또는 새 채팅에 첨부해, GuideFlow를 만들며 쌓은 개발 방식과 AI 협업 원칙을 빠르게 전달하기 위한 압축 문서입니다.

## 1. 프로젝트 목적

GuideFlow는 사용자가 여러 장면을 만들고, 장면 안의 이미지와 요소를 직접 조작하며, 순서대로 안내 콘텐츠를 구성할 수 있도록 만드는 웹 기반 편집 도구입니다.

이 프로젝트에서 중요한 것은 새로운 기술을 많이 사용하는 것이 아니라, 비개발자도 실제로 사용할 수 있는 기능을 작은 단위로 안정적으로 완성하는 것입니다.

## 2. 사용자와 AI의 역할

- 사용자는 제품의 목적, 사용 느낌, 우선순위와 최종 판단을 담당합니다.
- AI는 저장소 확인, 기술 설계, 코드 수정, 오류 분석, 문서화와 검증 방법 안내를 담당합니다.
- 사용자는 비개발자이므로 전문용어를 먼저 던지지 말고, 쉬운 설명과 비유를 먼저 제공해야 합니다.
- AI가 제품 방향을 임의로 확대하지 않습니다. 사용자가 요청한 범위 안에서 가장 작은 변경을 먼저 만듭니다.

## 3. 반드시 지킬 작업 순서

1. 현재 저장소와 관련 파일을 먼저 확인합니다.
2. 사용자가 요청한 기능을 한 문장으로 다시 정의합니다.
3. 한 번에 기능 하나 또는 확인 가능한 작은 범위만 수정합니다.
4. 기존 기능에 영향을 줄 수 있는 부분을 먼저 설명합니다.
5. 코드를 수정하고 GitHub 커밋을 확인합니다.
6. 배포 상태를 별도로 확인합니다.
7. 브라우저 동작을 직접 확인하지 못했다면 확인하지 못했다고 명확히 말합니다.
8. 사용자가 실제 화면을 확인한 뒤 다음 작업으로 넘어갑니다.

## 4. 완료와 검증의 기준

다음 상태를 서로 구분합니다.

- 구현됨: 코드가 저장소에 작성되었습니다.
- 커밋 확인됨: 변경이 GitHub 커밋에 포함되었습니다.
- 배포 확인됨: Vercel 등 배포 서비스가 성공 상태를 반환했습니다.
- 브라우저 검증됨: 실제 배포 주소에서 기능을 눌러 보고 기대한 결과를 확인했습니다.

브라우저에서 직접 확인하지 않았다면 “정상 동작한다” 또는 “완료됐다”고 단정하지 않습니다.

## 5. GuideFlow에서 사용한 핵심 기술

### HTML
화면의 구조를 만듭니다. 버튼, 장면 목록, 편집 영역, 이미지 입력과 같은 요소의 뼈대입니다.

### CSS
화면 배치, 크기, 간격, 반응형 모바일 화면과 선택 상태를 표현합니다.

### JavaScript
사용자가 버튼을 누르거나 이미지를 움직였을 때 실제 동작을 처리합니다. 장면 추가, 삭제, 복제, 선택, 이미지 교체와 저장 로직을 담당합니다.

### DOM
JavaScript가 화면의 HTML 요소를 찾고 변경할 수 있게 하는 브라우저 구조입니다. 장면 목록을 다시 그리거나 선택된 요소의 속성을 바꿀 때 사용합니다.

### Event Listener
클릭, 입력, 드래그 같은 사용자 행동을 감지합니다. 버튼과 입력 요소를 기능에 연결합니다.

### Pointer Event
마우스, 손가락 터치, 펜 입력을 하나의 방식으로 처리합니다. 이미지나 화면 요소의 드래그 조작에 적합합니다.

### 장면 데이터와 고유 ID
장면의 화면 순번과 내부 고유 ID를 분리합니다. 장면을 삭제하거나 이동해도 다른 장면을 잘못 가리키지 않게 합니다.

### 이미지 교체
기존 이미지 요소의 위치와 크기는 유지하면서 이미지 내용만 바꿉니다. 교체 전에 현재 선택된 이미지가 맞는지 확인해야 합니다.

### LocalStorage
브라우저 안에 작은 설정이나 작업 상태를 저장합니다. 큰 이미지나 중요한 영구 데이터 저장에는 적합하지 않습니다.

### Git과 GitHub
원본 코드와 변경 기록을 저장합니다. 문제가 생겼을 때 어떤 변경에서 발생했는지 비교하고 이전 상태를 확인할 수 있습니다.

### Vercel
GitHub의 웹 프로젝트를 실제 인터넷 주소로 배포합니다. GitHub 커밋 성공과 실제 배포 성공은 별도로 확인해야 합니다.

## 6. GuideFlow에서 반복적으로 중요했던 설계 원칙

- 화면에 보이는 장면 번호와 데이터 고유 ID를 같은 값으로 사용하지 않습니다.
- 현재 선택된 장면과 현재 선택된 요소를 명확히 관리합니다.
- 이미지 교체는 새 요소를 만드는 작업과 구분합니다.
- 드래그 시작, 이동, 종료 상태를 분리합니다.
- 기능 추가 전에 기존 동작이 무엇인지 먼저 확인합니다.
- 한 번에 큰 구조를 다시 만들기보다 기존 구조에서 가장 작은 수정을 우선합니다.
- 모바일과 PC 입력을 함께 고려합니다.
- 저장 기능은 데이터 크기와 영구성 요구를 먼저 판단한 뒤 선택합니다.

## 7. AI에게 기능을 요청하는 기본 형식

아래 형식을 사용합니다.

```text
현재 프로그램: [프로그램 이름]
현재 상태: [지금 보이는 동작]
추가하거나 수정할 기능: [기능 하나]
기대 결과: [사용자가 확인할 수 있는 결과]
수정 범위: [알고 있다면 파일 또는 화면]
보호할 기존 기능: [깨지면 안 되는 기능]
검증 방법: GitHub 반영, 배포 상태, 브라우저 확인을 구분해서 알려줘.
```

## 8. 오류 수정을 요청하는 형식

```text
현재 현상: [실제로 보이는 문제]
기대 결과: [원래 되어야 하는 동작]
재현 순서: [문제가 발생하는 클릭 순서]
최근 변경: [알고 있다면 작성]

추측으로 수정하지 말고 관련 코드를 먼저 확인해.
원인을 설명한 뒤 가장 작은 수정만 적용해.
수정 후 내가 직접 확인할 테스트 순서도 알려줘.
```

## 9. AI가 방향을 과도하게 확대할 때 사용할 문장

```text
새로운 구조나 기능을 제안하지 말고 지금 요청한 범위만 처리해.
현재 제품의 목적을 바꾸지 말고 가장 작은 수정으로 보여줘.
기획 설명보다 내가 실제로 확인할 수 있는 화면 변경을 먼저 만들어줘.
```

## 10. 새 프로젝트에 재사용할 원칙

- 목적을 한 문장으로 정합니다.
- 첫 버전은 핵심 기능 1~3개만 만듭니다.
- GitHub를 원본으로 사용합니다.
- 배포 결과와 브라우저 동작을 구분해 확인합니다.
- 작업을 작은 단위로 반복합니다.
- 실패 원인과 해결 방법을 문서로 남깁니다.
- 특정 도구보다 문제 해결 방식과 검증 습관을 재사용합니다.

## 11. 새 채팅에서 AI에게 함께 전달할 문장

```text
첨부한 GUIDEFLOW_CONTEXT.md를 먼저 읽고 그 규칙을 작업 기준으로 사용해.
나는 비개발자이므로 쉬운 말로 설명해.
요청 범위를 임의로 확대하지 말고 작은 단위로 구현해.
GitHub 반영, 배포 성공, 브라우저 동작 확인을 서로 구분해서 보고해.
확인하지 못한 내용은 완료됐다고 말하지 마.

이번에 만들 프로그램의 목적:
[여기에 한두 문장으로 작성]

지금 가장 먼저 만들 기능:
[기능 하나만 작성]
```
