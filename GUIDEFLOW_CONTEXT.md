# GuideFlow 개발 컨텍스트

이 문서는 새 ChatGPT, Codex 또는 Claude 작업에 첨부해 GuideFlow의 목적과 개발 경험을 한 번에 전달하는 압축 문서입니다.

## 1. GuideFlow가 해결하려는 문제

사용자는 영상 아이디어가 있어도 기존 편집 프로그램을 켜는 순간 작업이 커지고 무겁게 느껴져 시작을 미루는 경우가 많습니다.

GuideFlow의 목적은 완성 영상을 편집하는 것이 아닙니다.

> 인터넷에서 찾은 이미지와 캡처를 붙여 넣고 순서와 시간을 조정해, 약 5분 안에 영상 아이디어의 전체 느낌을 스케치하는 가벼운 가이드 편집 도구

GuideFlow로 만든 가이드는 나중에 실제 편집 프로그램을 열었을 때 무엇을 만들지 다시 고민하지 않게 해주는 영상 설계도 역할을 합니다.

## 2. 핵심 작업 흐름

1. 영상 아이디어가 떠오릅니다.
2. 인터넷에서 참고 이미지나 장면을 찾습니다.
3. 화면을 캡처하고 GuideFlow에 Ctrl+V로 붙여 넣습니다.
4. 이미지의 순서, 위치, 크기와 장면별 시간을 빠르게 조정합니다.
5. 전체 보드와 재생으로 흐름을 확인합니다.
6. 가이드를 저장하거나 내보냅니다.
7. 나중에 실제 편집 프로그램에서 가이드를 보며 본편집합니다.

성공 기준은 기능 수가 아니라 사용자가 부담 없이 열어 실제 편집의 시작 장벽을 낮추는 것입니다.

## 3. 제품 원칙

- 영상 편집 프로그램을 다시 만들지 않습니다.
- 기능보다 빠른 접근과 가벼운 사용감을 우선합니다.
- 사용자가 요청하지 않은 기능을 임의로 늘리지 않습니다.
- 설명보다 사용자가 확인할 수 있는 결과물을 먼저 만듭니다.
- 보드에서 전체 장면을 보며 바로 수정할 수 있어야 합니다.
- 이미지 추가와 이미지 교체를 명확히 구분합니다.
- PC와 모바일에서 버튼과 조작이 깨지지 않아야 합니다.
- 새 기술 도입은 실제 문제를 해결할 때만 검토합니다.

## 4. 실제 개발에서 중요했던 기능

### 이미지 붙여넣기
캡처 후 Ctrl+V를 누르면 이미지가 새 장면이나 선택한 위치에 바로 들어가는 기능입니다. 가져오기 창을 여러 번 거치지 않는 것이 핵심입니다.

### 붙여넣기로 이미지 교체
이미지를 새로 추가하는 것과 기존 이미지를 교체하는 것은 다른 동작입니다. 선택된 이미지가 있을 때 붙여넣으면 위치와 크기를 유지하고 내용만 교체할 수 있어야 합니다.

### 이미지 위치와 크기 조정
별도의 복잡한 편집 화면보다 전체 보드를 보면서 이미지를 직접 조절하는 방식이 더 빠르고 직관적입니다.

### 장면 관리
장면 추가, 삭제, 복제와 순서 변경이 필요합니다. 화면에 보이는 순번과 내부 고유 ID를 분리해야 순서를 바꾼 뒤에도 정확한 장면을 수정할 수 있습니다.

### 장면별 시간과 재생
각 장면의 표시 시간을 설정하고 순서대로 재생합니다. 특히 마지막 장면도 설정한 시간만큼 유지된 뒤 종료되어야 합니다.

### 내보내기
GuideFlow의 결과를 다른 사람에게 보여주거나 실제 편집 때 참고할 수 있는 형태로 만듭니다. 내보내기 결과는 편집 화면과 별도로 직접 확인해야 합니다.

### 모바일 UI
버튼 글자가 밖으로 튀어나오거나 배열이 무너지지 않아야 합니다. 작은 화면에서는 기능을 늘리는 것보다 핵심 버튼이 잘 보이는 것이 중요합니다.

### GitHub와 Vercel
GitHub는 원본과 변경 기록, Vercel은 실제 공개 결과입니다. 파일 수정, 커밋, 배포 성공과 브라우저 동작을 서로 다른 단계로 확인합니다.

## 5. 반복해서 발생한 문제와 교훈

### 개발 기술을 계속 바꾸는 문제
Tauri, Electron, React 등 기술 선택이 반복해서 바뀌면 사용자는 환경 설정만 하다가 지칩니다.

교훈: 가장 빠르게 확인 가능한 방식으로 첫 결과물을 만들고, 기술 변경은 현재 방식으로 해결할 수 없는 문제가 생겼을 때만 검토합니다.

### 사용자에게 설치와 오류 해결을 떠넘기는 문제
비개발자에게 여러 개발 도구 설치, 명령어 입력과 코드 복사를 반복해서 요구하면 프로젝트가 중단되기 쉽습니다.

교훈: AI가 직접 할 수 있는 저장소 확인, 코드 수정, 검사, 커밋과 배포 확인은 AI가 수행합니다.

### 보드와 편집 화면이 나뉘어 복잡해지는 문제
별도의 편집 모드로 이동하면 전체 장면의 흐름을 보면서 수정하기 어렵습니다.

교훈: 가능하면 메인 보드에서 선택과 조정을 함께 처리합니다.

### 마지막 장면 시간이 적용되지 않는 문제
다음 장면으로 넘어가는 로직만 생각하면 마지막 장면은 다음 대상이 없어 즉시 끝날 수 있습니다.

교훈: 마지막 장면의 유지 시간과 전체 재생 종료를 별도로 처리하고 내보내기 결과까지 확인합니다.

### 배포됐지만 실제 화면이 바뀌지 않는 문제
수정한 파일과 실제 빌드에 사용되는 파일이 다르면 커밋과 배포가 성공해도 사이트는 바뀌지 않습니다.

교훈: 빌드 진입 파일과 배포 결과물을 먼저 확인하고 실제 공개 주소에서 화면을 검증합니다.

## 6. 사용자와 AI의 역할

### 사용자
- 해결하려는 문제와 원하는 사용 느낌을 설명합니다.
- 실제로 사용해 보고 불편함을 판단합니다.
- 우선순위와 최종 완료 여부를 결정합니다.

### AI
- 저장소와 현재 동작을 먼저 확인합니다.
- 요청 범위를 임의로 확대하지 않습니다.
- 코드 작성, 오류 분석, 검증과 문서화를 담당합니다.
- 비개발자에게는 쉬운 말로 결과와 선택지만 설명합니다.
- 직접 할 수 있는 작업을 사용자에게 떠넘기지 않습니다.

## 7. 작업과 검증 순서

1. 현재 저장소와 실제 배포 구조를 확인합니다.
2. 이번에 바꿀 기능 하나와 유지할 기능을 정합니다.
3. 가장 작은 안전한 변경을 구현합니다.
4. 관련 검사를 실행합니다.
5. GitHub 커밋을 확인합니다.
6. 배포 상태를 확인합니다.
7. 실제 공개 화면과 기능을 직접 확인합니다.
8. 새로 얻은 기술과 시행착오를 문서에 반영합니다.

다음 네 상태를 구분해서 보고합니다.

- 구현됨
- GitHub 커밋 확인됨
- 배포 확인됨
- 브라우저 동작 확인됨
