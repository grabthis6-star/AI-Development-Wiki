# AI 협업 규칙

GuideFlow를 만들며 검증한, 다음 프로그램에도 그대로 적용할 수 있는 협업 규칙입니다.

## 핵심 규칙

### 💡 아이디어를 사용자 흐름으로 말하기
- 문제: “React로 타임라인을 만들어줘”처럼 기술부터 정하면 실제 목적과 다른 결과가 생깁니다.
- 규칙: 누가, 언제, 무엇을 누르고, 어떤 결과를 얻는지 먼저 말합니다.
- 좋지 않은 예시: 영어 공부 앱을 만들어줘.
- 개선된 예시: 학습자가 영상에서 들은 영어 문장을 붙여 넣으면 뜻·핵심 표현·예문을 한 화면에서 확인하는 웹 도구를 원해.
- AI 요청문:

> 나는 [사용자]가 [상황]에서 [행동]하면 [결과]를 얻는 프로그램을 원해. 필요한 기술은 네가 판단하되 먼저 가장 작은 결과물을 만들어.

### 🎯 첫 버전의 완료 기준 정하기
- 문제: 완성의 뜻이 다르면 AI는 기능을 계속 늘리고 사용자는 결과를 보지 못합니다.
- 규칙: “화면에서 무엇이 되면 끝인가”를 한 문장으로 정합니다.
- 좋지 않은 예시: 영어 공부에 필요한 기능을 전부 만들어.
- 개선된 예시: 영어 문장 하나를 입력하면 뜻과 예문이 화면에 표시되면 첫 버전은 완료야.
- AI 요청문:

> 이번 완료 기준은 [실제 화면에서 확인할 결과]야. 그 밖의 기능은 추가하지 마.

### 📦 AI에게 현재 프로젝트 넘기기
- 문제: 새 AI는 이전 대화와 시행착오를 자동으로 알지 못합니다.
- 규칙: 긴 채팅 대신 GUIDEFLOW_CONTEXT.md 한 개와 이번 작업을 전달합니다. 새 프로젝트에 없는 문서를 읽으라고 요구하지 않습니다.
- 좋지 않은 예시: 전에 하던 영상 도구 계속 만들어줘.
- 개선된 예시: PROJECT_CONTEXT.md를 읽고 현재 장면 보드가 동작하는지 확인한 뒤 장면 복제 기능만 추가해.
- AI 요청문:

> GUIDEFLOW_CONTEXT.md를 먼저 읽고 현재 폴더를 확인한 뒤 이번 작업만 수행해.

### 🧰 AI가 할 일과 사용자가 할 일 나누기
- 문제: AI가 코드 복사·파일 수정·불필요한 설치를 사용자에게 넘기면 초보자는 지칩니다.
- 규칙: AI가 접근 가능한 파일 수정·검사·커밋·배포 확인은 AI가 직접 합니다.
- 좋지 않은 예시: 사용자에게 수정할 코드와 터미널 명령을 길게 전달합니다.
- 개선된 예시: AI는 파일 수정과 검사를 직접 하고, 사용자는 공개 화면에서 사용감을 판단합니다.
- AI 요청문:

> 네가 직접 할 수 있는 코드 수정과 검증은 직접 해. 내가 꼭 해야 하는 선택이나 로그인만 쉬운 단계로 요청해.

### 🪜 한 번에 기능 하나만 맡기기
- 문제: 큰 요청은 기존 기능을 깨고 잘못된 방향을 늦게 발견하게 합니다.
- 규칙: 기능 하나 → 실제 확인 → 피드백 → 다음 기능 순서를 지킵니다.
- 좋지 않은 예시: 장면 생성·음성·자막·영상 출력까지 한 번에 만들어.
- 개선된 예시: 이번에는 아이디어를 장면 목록으로 바꾸는 기능만 만들고 화면에서 확인하자.
- AI 요청문:

> 이번에는 [기능 하나]만 구현해. 완료 후 내가 확인할 위치와 방법을 알려줘.

### 📸 문제는 증거와 함께 전달하기
- 문제: 짧은 증상 설명만으로는 AI가 원인을 추측합니다.
- 규칙: 오류 메시지는 생략하지 않고 전체를 복사하고 현재 화면을 첨부합니다.
- 좋지 않은 예시: 저장 버튼이 안 돼.
- 개선된 예시: 문장을 입력하고 저장을 누르면 목록이 비어 있어. 첨부 화면과 콘솔 오류를 기준으로 같은 순서로 재현해.
- AI 요청문:

> 첨부 화면과 오류 전문을 기준으로 원인을 확인해. 같은 방법을 반복하지 말고 수정 후 재현 순서대로 검증해.

### 🛡️ 유지해야 할 기능 적기
- 문제: AI는 요청한 부분을 고치며 주변 구조까지 넓게 바꿀 수 있습니다.
- 규칙: 수정 전에 유지할 기능과 건드리지 않을 범위를 명시합니다.
- 좋지 않은 예시: 복습 기능을 추가해.
- 개선된 예시: 복습 목록만 추가하고 기존 문장 분석과 결과 복사 기능은 그대로 유지해.
- AI 요청문:

> [새 기능]만 수정하고 [유지 기능]은 그대로 보호해. 완료 후 두 기능을 모두 테스트해.

### ✅ 완료를 네 단계로 확인하기
- 문제: 배포 성공만으로 버튼과 모바일 화면이 정상이라고 판단할 수 없습니다.
- 규칙: 확인하지 않은 단계는 완료됐다고 말하지 않습니다.
- 좋지 않은 예시: 배포가 성공했으니 완료됐다고 보고합니다.
- 개선된 예시: 공개 주소에서 문장 입력·결과 표시·모바일 배치를 직접 확인한 뒤 각각 보고합니다.
- AI 요청문:

> 구현·GitHub 커밋·배포 성공·실제 브라우저 동작을 각각 확인하고 결과를 구분해서 보고해.

### 🧭 AI가 방향을 키울 때 되돌리기
- 문제: 가벼운 도구가 플랫폼·OS·플러그인 구조로 커지며 원래 목적을 잃을 수 있습니다.
- 규칙: 새 제안이 사용자의 현재 문제를 직접 해결하지 않으면 보류합니다.
- 좋지 않은 예시: 간단한 영어 표현 도구에 커뮤니티·결제·랭킹을 함께 추가합니다.
- 개선된 예시: 현재 목표를 문장 입력과 표현 이해로 다시 제한하고 나머지는 다음 단계로 보류합니다.
- AI 요청문:

> 범위를 넓히지 마. 대명제는 [초보자가 AI와 협업해 프로그램을 완성하는 것]이고, 지금은 [이번 결과]만 필요해.

## 상황별 요청 양식

### 새 프로그램 시작하기

아이디어를 기술 용어 없이 전달합니다.

```text

나는 [누구]를 위한 [프로그램]을 만들고 싶어.
사용 흐름은 [행동 → 결과]야.
첫 버전은 [핵심 기능 1~3개]만 필요해.
내가 설치하거나 선택해야 할 것만 이유와 함께 알려주고, 코드 작업은 직접 해.

```

### 설치가 필요한지 확인하기

불필요한 환경 설정을 막습니다.

```text

[도구 이름]이 이번 기능에 정말 필요한지 먼저 판단해.
필요한 이유, 사용되는 위치, 공식 설치 주소, 설치 확인 방법을 쉬운 말로 알려줘.
필요하지 않다면 설치시키지 마.

```

### 버그 수정하기

증상 대신 증거를 전달합니다.

```text

현재 화면: [스크린샷]
재현 순서: [1, 2, 3]
실제 결과: [현재 상태]
원하는 결과: [기대 상태]
오류 전문: [전체 복사]
추측하지 말고 원인을 확인한 뒤 수정하고 같은 순서로 검증해.

```

### 디자인 피드백 전달하기

예쁘게가 아니라 달라져야 할 점을 말합니다.

```text

첨부 화면에서 [위치]가 [현재 문제]야.
나는 [원하는 느낌/행동]을 원해.
[유지할 요소]는 그대로 두고 이 부분만 수정한 뒤 PC와 모바일을 확인해.

```

### 새 AI에게 프로젝트 넘기기

새 채팅에서도 같은 원칙을 유지합니다.

```text

먼저 GUIDEFLOW_CONTEXT.md를 읽어.
현재 프로젝트 폴더와 최근 변경을 확인해.
이번 작업은 [기능 하나]이고 [기존 기능]은 반드시 유지해.
완료 기준은 [화면에서 확인할 결과]야.

```
