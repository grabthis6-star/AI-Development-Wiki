# GuideFlow에서 배운 AI 협업 노하우

이 문서는 이상적인 AI 사용법을 설명하는 이론서가 아닙니다. 비개발자가 GuideFlow를 실제로 만들면서 겪은 실패를 다음 프로그램에서 반복하지 않기 위한 작업 규칙입니다.

## 1. AI가 기술 방향을 계속 바꾸지 못하게 하기

### 실제 문제
처음에는 Electron, 다음에는 Tauri, 다시 React 웹으로 방향이 바뀌었습니다. 그 과정에서 Node.js, Git, Rust와 Visual Studio Build Tools까지 설치 이야기가 이어졌고 결과물보다 환경 설정이 커졌습니다.

### 왜 문제가 되나
비개발자에게 기술 변경은 단순한 선택이 아닙니다. 새 설치, 새로운 오류와 다시 시작해야 한다는 부담으로 이어져 프로젝트 포기의 원인이 됩니다.

### 협업 규칙
- 현재 기술로 해결할 수 없는 문제가 확인되기 전에는 기술을 바꾸지 않습니다.
- “더 가볍다”, “더 전문적이다”만으로 새 기술을 선택하지 않습니다.
- 기술보다 오늘 확인할 수 있는 결과물을 우선합니다.

### 바로 쓰는 요청문
~~~text
현재 기술을 유지해.
다른 기술이나 대규모 구조 변경을 제안하지 마.
현재 방식으로 해결할 수 없는 구체적인 이유가 확인될 때만 선택지를 설명해.
~~~

## 2. 불필요한 설치와 수동 작업을 사용자에게 넘기지 않기

### 실제 문제
개발 도구 설치, 터미널 명령, 코드 복사와 오류 화면 전달이 반복되면서 사용자가 개발보다 환경 설정에 더 많은 에너지를 쓰게 됐습니다.

### 협업 규칙
AI가 저장소와 컴퓨터 환경에서 직접 안전하게 할 수 있는 작업은 직접 수행합니다.

- 파일과 저장소 확인
- 코드 작성과 수정
- 오류 로그 분석
- 빌드와 검사
- GitHub 커밋
- 배포 상태 확인
- 브라우저 동작 확인
- 기술 문서 업데이트

사용자에게는 AI가 할 수 없는 로그인, 제품 판단과 실제 사용 피드백만 요청합니다.

### 바로 쓰는 요청문
~~~text
네가 직접 할 수 있는 작업은 직접 처리해.
나에게 코드 복사, 불필요한 설치와 반복적인 오류 전달을 요구하지 마.
내 선택이 꼭 필요한 경우에만 쉬운 말로 물어봐.
~~~

## 3. 계획보다 확인 가능한 결과물을 먼저 만들기

### 실제 문제
사용자가 “진행해”라고 여러 번 말했지만 AI가 스프린트, OS, 플러그인과 미래 로드맵 설명을 계속 늘리고 실제 화면 변경을 하지 않는 일이 반복됐습니다.

### 협업 규칙
- 설명은 짧게 하고 결과물을 먼저 만듭니다.
- 한 번에 사용자가 확인할 수 있는 기능 하나를 완성합니다.
- 다음 계획은 현재 결과를 사용자가 본 뒤 세웁니다.
- “진행하겠다”가 아니라 변경 파일과 실제 화면으로 보고합니다.

### 바로 쓰는 요청문
~~~text
계획을 더 늘리지 말고 지금 요청한 결과물을 먼저 만들어.
내가 실제 화면에서 확인할 수 있는 상태까지 구현하고 검증해.
~~~

## 4. 요청 범위를 임의로 확대하지 않기

### 실제 문제
가벼운 영상 아이디어 도구가 편집 프로그램, AI Development OS, 학습 플랫폼과 플러그인 구조로 계속 커졌습니다.

### 협업 규칙
새 아이디어가 좋아 보여도 현재 목적에 직접 필요하지 않으면 구현하지 않습니다.

판단 질문은 하나입니다.

> 이 변경이 사용자가 5분 안에 영상 아이디어를 스케치하는 데 직접 도움이 되는가?

아니라면 보류합니다.

### 바로 쓰는 요청문
~~~text
GuideFlow의 목적을 바꾸지 마.
요청하지 않은 기능과 구조를 추가하지 말고 지금 기능 하나만 처리해.
좋은 아이디어가 있어도 구현하지 말고 짧게 메모만 남겨.
~~~

## 5. 기존 기능을 보호하며 수정하기

### 실제 문제
이미지 교체나 버튼 배치를 고치면서 이미 되던 붙여넣기, 장면 시간 또는 다른 버튼이 깨질 수 있습니다.

### 협업 규칙
1. 수정 전에 관련 파일과 현재 동작을 확인합니다.
2. 반드시 유지할 기능을 적습니다.
3. 가장 작은 범위만 바꿉니다.
4. 수정 기능과 기존 핵심 기능을 함께 테스트합니다.

### 바로 쓰는 요청문
~~~text
이번에는 [수정할 기능]만 바꿔.
[유지할 기능 목록]은 그대로 유지해야 해.
관련 코드를 먼저 확인하고 가장 작은 수정으로 처리한 뒤 기존 기능도 다시 테스트해.
~~~

## 6. 스크린샷과 재현 순서로 오류 전달하기

### 실제 문제
“버튼이 이상하다”, “파일 교체가 안 된다”만으로는 원인을 정확히 찾기 어렵습니다. 반대로 스크린샷, 현재 결과와 클릭 순서가 있으면 AI가 화면과 코드를 함께 비교할 수 있습니다.

### 사용자에게 필요한 정보
- 현재 보이는 화면 또는 스크린샷
- 원래 기대한 결과
- 문제가 생기는 클릭 순서
- 직전에 바뀐 내용
- PC인지 모바일인지

### 바로 쓰는 오류 양식
~~~text
현재 현상:
기대 결과:
재현 순서:
사용 환경:
최근 변경:
첨부 화면:

추측으로 수정하지 말고 관련 코드를 먼저 확인해.
원인을 설명하고 가장 작은 수정 후 확인 순서를 알려줘.
~~~

## 7. 구현·커밋·배포·실제 동작을 구분하기

### 실제 문제
파일을 수정했지만 실제 빌드가 다른 파일을 사용해 사이트가 바뀌지 않았습니다. 커밋과 Vercel 상태가 성공해도 버튼이 실제로 동작하지 않는 경우도 있었습니다.

### 네 단계
1. 구현됨: 코드가 작성됐습니다.
2. 커밋 확인됨: GitHub에 변경이 저장됐습니다.
3. 배포 확인됨: Vercel 등이 성공 상태를 반환했습니다.
4. 브라우저 검증됨: 실제 공개 주소에서 기능을 직접 사용했습니다.

### 바로 쓰는 요청문
~~~text
구현, GitHub 커밋, 배포 성공과 브라우저 동작을 각각 구분해서 보고해.
공개 주소에서 직접 확인하지 못한 기능은 완료됐다고 말하지 마.
~~~

## 8. 오류를 추측하지 말고 실제 원인을 찾기

### 실제 문제
버튼 배치와 마지막 장면 시간 오류가 “고쳤다”는 보고 뒤에도 남아 있었습니다. 코드 작성만 하고 실제 원인과 결과를 확인하지 않았기 때문입니다.

### 협업 규칙
- 증상을 가리는 CSS나 임시 예외부터 추가하지 않습니다.
- 관련 데이터 흐름과 실제 빌드 파일을 확인합니다.
- 같은 수정이 실패하면 같은 접근을 반복하지 않습니다.
- 수정 후 문제를 일으킨 정확한 순서를 다시 실행합니다.

### 바로 쓰는 요청문
~~~text
이전 수정이 실패했으니 같은 방법을 반복하지 마.
관련 코드와 실제 배포 파일을 다시 확인해서 근본 원인을 찾아.
수정 뒤에는 내가 말한 재현 순서를 직접 테스트해.
~~~

## 9. 새 AI에게 프로젝트 맥락 전달하기

### 실제 문제
새 채팅을 시작할 때마다 GuideFlow의 목적, 사용자의 수준과 이전 시행착오를 다시 설명해야 했습니다.

### 해결 방식
새 AI에게 다음 문서를 먼저 읽힙니다.

1. GUIDEFLOW_CONTEXT.md
2. TECH_INDEX.md
3. AI_RULES.md

그다음 이번 작업 하나와 유지할 기능만 전달합니다.

### 새 작업 시작문
~~~text
GUIDEFLOW_CONTEXT.md, TECH_INDEX.md, AI_RULES.md를 먼저 읽어.

나는 비개발자이고 제품의 사용 느낌과 우선순위를 판단한다.
너는 저장소 확인, 구현, 검사, 커밋, 배포 확인과 문서화를 담당한다.
요청 범위를 확대하거나 기술을 임의로 바꾸지 마.

이번 작업:
[기능 하나]

반드시 유지할 기능:
[기존 기능]

완료 기준:
[내가 실제로 확인할 결과]
~~~

## 10. 한 작업이 끝날 때 남길 기록

- 무엇을 바꿨는가
- 왜 필요했는가
- 실제 원인은 무엇이었는가
- 실패한 접근은 무엇이었는가
- 어떻게 검증했는가
- 다음 프로젝트에서 재사용할 규칙
- 관련 GitHub 커밋
