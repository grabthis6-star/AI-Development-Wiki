# AI 협업 규칙

GuideFlow에서 겪은 시행착오를 다음 프로그램에서도 반복하지 않기 위한 실전 규칙입니다.

## 가장 중요한 역할 분담

- 사용자는 문제, 원하는 사용 느낌, 우선순위와 최종 판단을 맡습니다.
- AI는 저장소 확인, 설계, 코드 수정, 오류 분석, 테스트와 기록을 맡습니다.
- 사용자는 개발자가 아니어도 됩니다. AI는 쉬운 말로 결과와 선택지만 설명해야 합니다.

## 작업을 시작할 때

1. AI는 관련 파일과 현재 동작을 먼저 확인합니다.
2. 요청을 임의로 더 큰 제품이나 새로운 기술로 확대하지 않습니다.
3. 이번 작업의 완료 기준을 사용자가 눈으로 확인할 수 있는 문장으로 정합니다.
4. 기존에 반드시 유지해야 할 기능을 확인합니다.
5. 가장 작은 안전한 변경부터 구현합니다.

## 구현 규칙

- 한 번에 거대한 구조를 다시 만들지 않습니다.
- 요청하지 않은 기능은 추가하지 않습니다.
- 기존 사용자 변경을 함부로 덮어쓰지 않습니다.
- 오류를 추측으로 고치지 않고 관련 코드와 재현 순서를 확인합니다.
- 기술 선택은 멋있어 보이는가가 아니라 목적에 필요한가로 판단합니다.
- 비개발자에게 불필요한 설치와 수동 복사 작업을 넘기지 않습니다.
- AI가 직접 안전하게 할 수 있는 코딩, 검사와 문서화는 AI가 처리합니다.

## 완료 보고 규칙

아래 네 상태를 반드시 구분합니다.

1. 구현됨: 코드가 작성되었습니다.
2. 커밋 확인됨: 변경이 GitHub에 저장되었습니다.
3. 배포 확인됨: 배포 서비스가 성공 상태를 반환했습니다.
4. 브라우저 검증됨: 실제 주소에서 기능을 직접 사용해 확인했습니다.

확인하지 않은 단계는 완료됐다고 말하지 않습니다.

## 좋은 기능 요청 형식

~~~text
현재 프로그램:
현재 보이는 동작:
이번에 바꿀 기능 하나:
사용자가 기대하는 결과:
반드시 유지할 기존 기능:
완료 후 확인할 방법:
~~~

## 좋은 오류 수정 요청 형식

~~~text
현재 현상:
원래 기대한 결과:
문제가 생기는 순서:
최근에 바뀐 내용:

관련 코드를 먼저 확인하고 원인을 설명해.
가장 작은 수정만 적용해.
수정 후 내가 확인할 순서도 알려줘.
~~~

## 범위가 커질 때 사용하는 문장

~~~text
새로운 제품이나 구조를 제안하지 말고 지금 요청한 범위만 처리해.
기획 설명보다 내가 확인할 수 있는 결과물을 먼저 만들어줘.
기존 기능은 유지하고 가장 작은 수정으로 진행해.
~~~

## 새 프로젝트 시작 프롬프트

~~~text
GUIDEFLOW_CONTEXT.md, TECH_INDEX.md, AI_RULES.md를 먼저 읽어.
나는 비개발자이므로 쉬운 말로 설명해.
내가 요청한 범위를 임의로 확대하지 마.
AI가 직접 할 수 있는 작업은 직접 처리하고, 나에게 불필요한 설치나 코드 복사를 요구하지 마.
구현, GitHub 커밋, 배포, 브라우저 검증을 구분해서 보고해.

이번 프로그램의 목적:
[한두 문장]

첫 번째로 필요한 기능:
[기능 하나]
~~~
